下面是基于 **方案 1：历史记录归档翻页区 + 最近记录热区** 的开发计划。这个方案与项目当前方向是连续演进，不是推翻重做：仓库已经有“两区模型”、归档区、固定批次、parking engine、本地数据管理这些基础，新的工作重点是把“固定批次展开”升级成“分页冷区”。 

## 一、目标定义

### 产品目标

在**不明显改变用户操作习惯**的前提下，让超长 ChatGPT 线程保持可用：

* 最近消息继续留在官方原生热区
* 更早历史进入插件控制的归档翻页区
* 冷区一次只渲染很少一页，降低 DOM 压力
* 视觉尽量贴近原生 ChatGPT
* 不依赖官方新增分页能力
* 保持本地优先、低权限、可回退

这与项目当前“保留原生 ChatGPT 流程、只缩小 live subtree”的定位一致。 

### 本期不做

这次先不把范围拉太大，以下内容延后：

* Auto / Fixed 热窗口模式
* 完全官方 UI 的“伪原生分页”
* 深层编辑 / regenerate / 原生菜单在冷区的完整复用
* 跨设备同步、导出、全文索引

这也符合当前架构文档里的非目标和保守策略。

---

## 二、总体方案

### 新的页面模型

从现在的：

**固定 hot transcript + 固定 slot batch archive**

升级为：

**最近记录热区 + 历史记录归档翻页区**

### 用户看到的交互

默认推荐做成：

* 页面底部到输入框附近，仍然是原生 ChatGPT 热区
* 热区上方出现一个轻量“历史边界”
* 边界上显示：

  * 当前历史页范围
  * 更早 / 更新
  * 返回最近
  * 页内消息数
* 点击后只加载一页冷历史
* 冷历史页仍在原对话流中，不跳转到新页面，不切到 reader mode

### 技术核心

* 热区仍保留当前模式，先固定常量
* 冷区改为“页”，不是“展开所有归档批次”
* 一次只挂载当前页，最多缓存相邻 1 页
* 更老历史以数据形式保存，尽量少长期持有 detached DOM
* 近端历史页可优先复用 host snapshot / 轻量 near-native 渲染

当前仓库已有 `ManagedHistoryStore`、`ParkingLot`、`StatusBar`、两区结构和 fixed-slot grouping，这些都能复用，只是职责要调整。

## 三、建议的里程碑

## Phase 0：设计冻结与技术探针

目标：先验证“分页冷区”不会破坏现有热区与停车逻辑。

### 输出

* 一份分页冷区设计文档
* 一份数据模型草案
* 一份 UI 线框草图
* 一个最小原型：只显示 1 页归档历史，不接搜索、不接高级恢复

### 关键决策

1. **页大小**

   * 建议先固定为 20 对 interaction pairs
   * 后续再看是否允许配置

2. **热区大小**

   * 继续沿用当前固定值
   * 本期不做 Auto / Fixed

3. **页缓存策略**

   * 当前页常驻
   * 相邻页可选缓存 1 页
   * 更远页销毁 DOM，仅保留数据

4. **冷区渲染策略**

   * 优先用现有 near-native 渲染器
   * 对能安全保留的内容继续支持 host snapshot

### 验收标准

* 能在不改现有热区核心逻辑的前提下，插入一个分页冷区容器
* 能从已有历史数据中切出“第 N 页”

---

## Phase 1：数据层重构

目标：让“归档历史”从“固定批次集合”升级成“可分页历史集合”。

### 需要改造的核心模块

#### 1. `ManagedHistoryStore`

当前它已经能：

* 维护 turns
* 生成 pairs
* 生成 archive groups
* 做搜索匹配
* 处理 initial trim 和 live records 合并 

需要新增：

* `getArchivedPairsWindow(pageIndex, pageSize, hotPairCount)`
* `getArchivedPageCount(pageSize, hotPairCount)`
* `getArchivedPageMeta(pageIndex, pageSize, hotPairCount)`
* `findPageIndexByTurnId(...)`
* `findPageIndexByQueryMatch(...)`

#### 2. 新增 `ArchivePager`

建议单独建一个分页状态层，不把分页状态塞进 `ManagedHistoryStore`。

职责：

* 当前页 index
* 总页数
* 页缓存
* 页跳转
* 页边界计算
* 搜索跳页映射

### 数据结构建议

```ts
type ArchivePage = {
  id: string;
  pageIndex: number;
  pairStartIndex: number;
  pairEndIndex: number;
  pairCount: number;
  entries: ManagedHistoryEntry[];
  source: 'initial-trim' | 'parked-dom' | 'mixed';
}
```

### 验收标准

* 能从完整历史稳定生成页
* 页切换不会影响热区 pair 计算
* initial-trim + parked-dom 混合历史仍可稳定分页

---

## Phase 2：UI 与渲染层替换

目标：把当前“批次卡片区”改造成“分页冷区”。

### 具体任务

#### 1. 新建分页边界条

建议放在热区上方，显示：

* `更早历史`
* `第 X / Y 页`
* `更早`
* `更新`
* `返回最近`

#### 2. 冷区只渲染当前页

当前 status bar / archive 区更多是围绕 batch card 展开。要改成：

* 不默认展示多个 archive groups
* 当前页内部可以按较小 batch 显示，避免单页太重
* 但页之间切换是唯一主交互

#### 3. 视觉风格

保持：

* 用户消息右对齐
* assistant 阅读流贴近原生
* markdown / code / table / quote 延续现有 near-native 风格

当前项目已经明确把冷区渲染做成“只读、近原生 transcript style”，不是卡片阅读器，这一点要保留。

### 验收标准

* 用户进入长对话时，默认只看到热区 + 一个轻量历史边界
* 点击历史翻页后，只出现当前页冷历史
* 来回翻页时滚动位置稳定，不跳屏

---

## Phase 3：Parking 与分页联动

目标：让停车引擎为“分页冷区”服务，而不是继续围绕旧批次 UI 工作。

### 关键改造

#### 1. `ParkingLot`

当前 `ParkingLot` 以 group 为单位 park/restore，并保留节点引用。

建议变成双层策略：

* **近端冷页**

  * 可以继续保留可恢复 DOM
* **远端冷页**

  * 不再长期保留节点数组
  * 只保留结构化数据 + snapshot

#### 2. `TurboRenderController`

当前控制器负责：

* 识别 live turn
* 计算 archive groups
* 决定 parking
* 更新 status UI
* 维护 batch expanded state 

需要改造为：

* 维护 `currentArchivePageIndex`
* 基于 page，而不是 expanded batch ids，更新冷区
* 将 `expandedArchiveBatchIds` 逐步迁移为：

  * 当前页 id
  * 页内局部展开状态

### 验收标准

* 页切换后，冷区不残留旧页大块 DOM
* 热区仍保持稳定
* 近端页翻回时可快速恢复，远端页则重建只读 DOM

---

## Phase 4：搜索与跳页

目标：保留“归档区本地搜索”，但从“命中批次”升级到“命中页”。

### 任务

* 搜索先在数据层查页，不直接依赖当前 DOM
* 搜索结果显示：

  * 命中页号
  * 摘要
  * 命中数量
* 点击搜索结果：

  * 自动切到对应页
  * 自动滚动到命中位置
  * 页内高亮

当前 `ManagedHistoryStore.search` 已经有 excerpt 和 match count 基础，可复用。

### 验收标准

* 搜索冷历史不需要先把所有页渲染出来
* 命中跳页后高亮准确
* 搜索不会导致冷区一次性挂载所有页

---

## Phase 5：测试、性能回归、灰度

目标：确认新方案在真实长线程下比当前 batch 方案更稳。

### 测试分层

#### 单元测试

* 分页边界计算
* 页范围计算
* 页跳转
* 搜索命中映射到页
* initial-trim + parked-dom 混合分页

#### 集成测试

* 热区保留最近 N 对
* 冷区只挂当前页
* 切页时滚动保持
* 切页不导致热区重建
* 恢复最近页 / 远端页行为差异

#### 手工验证

* `/c/<id>` 长线程
* `/share/<id>` 长线程
* 代码块密集线程
* 表格/quote/长 markdown 线程
* 正在 streaming 的线程

当前仓库已有 unit、integration、Playwright 和 controlled Chrome 调试流程，这能直接承接。 

### 关键指标

建议加三类观察值：

* live descendants 数
* 冷区页切换耗时
* frame spike 次数
* 输入延迟主观回归
* 内存峰值对比旧方案

## 六、主要风险与规避

### 风险 1：页切换导致滚动跳变

规避：

* 以“边界锚点”保持滚动
* 切页时只替换冷区内部，不改热区位置

### 风险 2：冷区页内容太重

规避：

* 页大小先保守，建议 20 对
* 页内必要时二级分块渲染

### 风险 3：保留太多 detached DOM，内存仍高

规避：

* 近页可恢复，远页数据化
* 页缓存最多当前页 ±1

### 风险 4：过度依赖宿主 DOM

规避：

* 热区继续依赖宿主
* 冷区尽量用结构化渲染 + 可选 snapshot
* 不把“官方假分页”当主线

---

## 七、我建议的首版验收标准

首版上线前，至少满足这些：

1. 长线程进入页面后，默认只显示热区和轻量历史边界
2. 冷区一次只渲染 1 页
3. 翻页不会明显跳屏
4. 最近消息的原生交互不受影响
5. 代码块和 markdown 在冷区可读性不退化
6. 比当前“多批次展开”模式更稳，至少在滚动和输入上主观更顺滑
7. 出现宿主不兼容时，可直接回退到现有方案或更保守模式

---

## 八、最后建议

这次开发计划建议明确一个原则：

**先把“分页冷区主链路”做稳，再谈热窗口智能化。**

也就是先完成：

* 冷区分页
* 近原生渲染
* 页切换稳定
* DOM 预算可控

等这一版稳定后，再把 Auto / Fixed 热窗口模式加进去，会更安全，也更容易看出收益。

下一步我可以把这份计划继续细化成：
**GitHub Projects / Jira 可直接执行的 Epic → Story → Task 看板版本**。
