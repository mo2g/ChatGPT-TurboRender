# 基于 CDP 连接已登录 Chrome for Testing 的开发与自动测试指南

这份文档定义新的主线方案：

**不再把离线环境作为 TurboRender 的主开发路径。后续开发、联调、自动测试，统一以 `CDP + 已登录 Chrome for Testing + 真实 chatgpt.com` 为准。**

这不是推翻重做，而是回到仓库已经具备的稳定链路：受控浏览器、固定调试端口、隔离 profile、已登录真实页面、Playwright 通过 CDP 附着执行自动测试。

## 一、背景与决策

前一阶段我们评估过“离线回放 / fixture / mock server”路线。它能提供一定的确定性，但对这个项目来说，它不是最稳的主线，原因很直接：

* TurboRender 的核心问题发生在 **真实 ChatGPT 宿主环境** 中。
* 插件依赖官方页面的 DOM、交互轨道、消息标识、菜单行为和登录态。
* 一旦离线壳、接口回放、静态资源版本与官方产生偏差，就会出现“离线通过、线上失效”的错觉。
* 这类错觉对当前项目的伤害，比“少一点测试确定性”更大。

因此，新主线改为：

* 开发时直接附着到已登录的 `Chrome for Testing`
* 页面保持真实 `https://chatgpt.com/...`
* 扩展以 unpacked 方式加载
* 自动测试以 CDP 连接该浏览器执行
* 真实页面行为作为最终事实来源

## 二、目标定义

### 工程目标

1. 让开发环境与官方 ChatGPT 尽可能一致。
2. 让“本地开发 -> 扩展重载 -> 真实页验证 -> 自动测试”形成稳定闭环。
3. 把宿主兼容性问题尽早暴露在真实页面上，而不是推迟到最后。
4. 保留现有受控浏览器脚本与测试基础设施，减少额外系统复杂度。

### 本期不做

这次不把范围扩大到以下方向：

* 不再投入新的“离线等价环境”建设
* 不要求离线 fixture 继续作为主验收门
* 不在本期删除所有离线脚本和文档
* 不尝试绕过真实登录态，或保存完整用户凭证到仓库
* 不承诺无人值守的公有 CI 直接跑登录态 ChatGPT 真实页回归

离线能力如果保留，也只作为辅助工具或历史资产，不再作为主设计前提。

## 三、主线开发模型

新的开发模型分成三层：

### 1. 真实宿主层

* 浏览器：`Chrome for Testing` 或可控 Chromium
* 页面：真实 `chatgpt.com`
* 登录态：开发者手工登录，保存在隔离 profile
* 事实来源：官方 DOM、真实接口、真实交互

### 2. 扩展开发层

* 扩展通过 `.output/chrome-mv3` 以 unpacked 方式加载
* 修改代码后重新 `build`
* 使用受控浏览器内的扩展重载机制刷新，而不是频繁销毁登录浏览器

### 3. 自动测试层

* Playwright 不自行新建“空白浏览器世界”
* Playwright 通过 `connectOverCDP()` 附着到已运行、已登录的浏览器
* 测试目标是真实 ChatGPT 页面，而不是本地 mock 页面

这三层共享同一个宿主真相，因此更适合 TurboRender 这类强依赖宿主页面语义的插件。

## 四、推荐的日常开发链路

### Step 1：构建扩展

```bash
pnpm build
```

首次启动前必须有 `.output/chrome-mv3`。

### Step 2：启动受控浏览器

```bash
pnpm debug:mcp-chrome -- https://chatgpt.com/
```

或直接打开目标会话：

```bash
pnpm debug:mcp-chrome -- https://chatgpt.com/c/ceb4ea77-5357-49fb-b35c-607b533846f1
```

这一步的职责是：

* 启动受控的 Chrome for Testing / Chromium
* 固定远程调试端口，默认 `9222`
* 加载 unpacked 扩展
* 使用隔离 profile 保存登录态

### Step 3：在受控浏览器中登录

这一步只做一次，后续复用该 profile。

要求：

* 保持该浏览器窗口和 profile 持续可用
* 不要把这个 profile 当成日常浏览器乱用
* 最好使用单独开发账号或最小风险账号

### Step 4：让 Codex / Playwright 附着到该浏览器

仓库级 `.codex/config.toml` 已经把 `chrome-devtools` MCP 指向 `http://127.0.0.1:9222`。

如果刚启动了受控浏览器，按现有 cookbook 的建议重开一次 Codex 会话，让 MCP 明确附着到这个实例。

推荐立刻执行一次：

```bash
pnpm check:mcp-chrome
```

它会直接检查调试端口、精确匹配当前默认的长对话 `/c/...` 目标页签，并确认 TurboRender 注入标记和 archive 就绪状态是否已经出现。

### Step 5：修改代码后重建并热重载扩展

```bash
pnpm build
pnpm reload:mcp-chrome
```

这样会：

* 触发 `chrome://extensions` 内的扩展重载
* 刷新当前 ChatGPT 标签页
* 尽量保留原浏览器和登录态

### Step 6：在真实页面上验证

优先检查：

* TurboRender 注入根节点是否存在
* 热区 / 归档区是否按预期工作
* 宿主原生按钮与插件交互是否仍对齐
* 滚动、输入、菜单、朗读等高风险路径是否受影响

如果扩展没有注入，优先查看 [docs/cookbook-controlled-chrome.zh-CN.md](../cookbook-controlled-chrome.zh-CN.md)。

## 五、自动测试策略

新主线下，测试不再追求“全部离线可复现”，而是追求“真实宿主下尽早发现兼容性问题”。

### 测试分层

#### 1. 单元测试

职责：

* 纯数据逻辑
* 算法边界
* 非宿主依赖的工具函数

命令：

```bash
pnpm test:unit
```

#### 2. 真实页自动测试

职责：

* 检查扩展是否成功注入真实 ChatGPT
* 检查关键宿主交互是否仍成立
* 检查 TurboRender 与官方 UI 的耦合点是否回归

命令：

```bash
pnpm test:e2e
pnpm test:e2e -- --chat-url=https://chatgpt.com/c/ceb4ea77-5357-49fb-b35c-607b533846f1
```

如果你明确要用当前活动标签页作为目标，而且能确认它已经是目标长对话：

```bash
pnpm test:e2e -- --use-active-tab
```

`pnpm test:e2e:live` 仍保留为同一套 live runner 的显式别名。扩展自有界面不再单独维持浏览器 E2E 车道，宿主兼容性证据统一回到真实 ChatGPT 页面。

#### 3. 手工 smoke 验证

职责：

* 覆盖自动测试暂时未覆盖的真实阅读链路
* 验证肉眼可感知的性能与滚动稳定性
* 处理时序敏感、菜单层级、流式变化等问题

推荐最少覆盖：

* 长对话 `/c/<id>`
* 代码块密集对话
* 含表格 / quote / 长 markdown 的对话
* 涉及 Read Aloud 的对话

### 当前现实与后续约束

当前仓库里：

* `pnpm test:e2e` 已切到主线真实页 smoke，并默认使用 `https://chatgpt.com/c/ceb4ea77-5357-49fb-b35c-607b533846f1`
* `pnpm test:e2e:live` 保留为同一套 smoke 的显式别名
* popup 等扩展自有界面留在宿主 E2E 之外，继续通过单元/集成测试和手工检查保障
* 旧离线 fixture 捕获、检查和假宿主回放工具已从当前有效测试面移除

因此本次需求的一个明确工作项是：

**把“真实页 CDP 自动测试”从补充路径提升为默认开发验收路径。**

## 六、开发约束

为了让这条路线长期可维护，后续开发必须遵守这些约束：

### 1. 官方页面是真相源

如果真实页面与离线 / mock / fixture 表现不一致：

* 以真实页面为准
* 修真实页回归
* 不为了保离线兼容而牺牲主线行为

### 2. 新增 E2E 用例统一走受控浏览器封装

不要在每个 spec 里各自手写一套 CDP 附着逻辑。统一复用：

* `tests/e2e/controlled-browser.ts`

这样才能统一：

* 调试端口
* profile 复用
* 外部浏览器清理策略
* “require-existing / prefer-existing / managed” 模式

### 3. 不把登录态导出成仓库资产

明确禁止：

* 提交 cookies
* 提交 storageState
* 提交 bearer token
* 提交任何可直接复用的认证信息

### 4. 真实页测试只断言关键语义，不断言脆弱细节

要优先断言：

* 注入是否成功
* 必要控件是否存在
* 菜单 / 操作是否能走通
* 关键 DOM 锚点是否可定位

要谨慎断言：

* 过细的像素位置
* 容易漂移的类名
* 短期会变化的实验性文案

### 5. 真实测试目标要长期稳定

后续需要维护一组稳定的测试目标：

* 专门的长对话链接
* 专门的分享页链接
* 覆盖代码块 / markdown / read aloud 的样本会话

测试目标不稳定，live 自动测试就会持续失真。

## 七、建议的里程碑

这部分沿用 [docs/plan/1.md](./1.md) 的写法：按阶段推进，每阶段都有明确产物与验收标准。

## Phase 0：路线冻结与文档切换

目标：正式把主线从“离线优先”切回“CDP 真实页优先”。

### 输出

* 一份新的主线开发指南
* 一份新的实施计划
* 对团队口径的统一说明

### 关键任务

1. 写清主线方案与非目标
2. 明确 `pnpm debug:mcp-chrome` / `pnpm reload:mcp-chrome` / `pnpm test:e2e` 的职责
3. 明确离线路径不再作为主验收门

### 验收标准

* 新人只看文档就能完成启动、登录、重载、验证
* 团队对“主开发链路是什么”不再有歧义

## Phase 1：开发入口固化

目标：把当前受控浏览器链路从“可用”提升到“默认可依赖”。

### 需要落实的模块

#### 1. 受控浏览器启动链路

依托：

* `scripts/debug-mcp-chrome.mjs`

要求：

* 能稳定复用已存在的调试浏览器
* 能明确提示当前 profile 路径
* 能稳定加载 `.output/chrome-mv3`
* 能通过独立检查命令快速确认 endpoint 和页面注入状态

#### 2. 扩展重载链路

依托：

* `scripts/reload-mcp-chrome.mjs`

要求：

* 不要求关闭浏览器
* 不丢失登录态
* 能刷新当前 ChatGPT 页签

#### 3. 受控浏览器测试适配层

依托：

* `tests/e2e/controlled-browser.ts`

要求：

* 新测试统一复用
* 不允许零散复制 `connectOverCDP()` 逻辑

### 验收标准

* 从零开始的本地机器，按文档可以跑通一次完整开发链路
* 修改代码后，能在 1 次 build + reload 后看到真实页变化

## Phase 2：真实页自动测试基线

目标：让 live 自动测试承担主兼容性回归职责。

### 输出

* 一组最小但高价值的 live smoke 用例
* 一套稳定的测试目标会话
* 一份失败时的排障路径

### 建议任务

1. 以 `live-smoke.spec.ts` 为起点，整理出明确的 smoke 集
2. 为长对话、分享页、read aloud、归档区交互补齐核心断言
3. 把“扩展已注入”“宿主按钮仍可联动”做成复用 helper

### 验收标准

* 本地已登录受控浏览器上，核心 smoke 能稳定跑通
* 回归问题优先在 live 测试中暴露，而不是只在手工验证里发现

## Phase 3：主线工作流切换

目标：让开发者默认走真实页链路，而不是被旧的离线路径带偏。

### 建议任务

1. 更新相关文档，明确哪条路径是主线，哪条是历史辅助
2. 评估是否需要调整脚本命名，使 `live` 语义更直观
3. 在评审口径上，把 live 测试结果提升为主要证据

### 验收标准

* 新需求评审时，不再默认要求先补离线 fixture
* 日常开发和回归说明，以真实页验证为第一证据

## Phase 4：离线路径降级与隔离

目标：避免历史资产继续影响主线判断，但不急于一次性删除。

### 建议任务

1. 标记离线相关文档为“非主线 / 历史方案 / 辅助工具”
2. 评估哪些 fixture 测试仍有保留价值
3. 将需要保留的纯数据回归与宿主兼容性回归分开

### 验收标准

* 团队不会再把离线通过误判为真实页通过
* 离线路径即便保留，也不会再主导需求设计

## 十、主要风险与规避

### 风险 1：真实页测试存在波动

规避：

* 使用稳定测试会话
* 断言关键语义，不断言易漂移细节
* 保留必要超时与最小等待

### 风险 2：登录态失效影响开发节奏

规避：

* 使用隔离 profile
* 保持受控浏览器长期复用
* 尽量减少销毁重建浏览器的次数

### 风险 3：开发者误连到错误浏览器实例

规避：

* 固定调试端口 `9222`
* 明确受控浏览器启动脚本为唯一推荐入口
* 出问题时先检查 `http://127.0.0.1:9222/json/version`

### 风险 4：真实页自动测试难以直接进公有 CI

规避：

* 接受“本地 live 回归 + 单元测试 CI”作为阶段性现实
* 如果未来要做持续自动化，再单独引入受控账号、自托管 runner、隔离密钥体系

### 风险 5：历史离线路径继续误导判断

规避：

* 明确标注其定位
* 不再把其结果当主通过条件
* 在新文档和评审说明中统一口径

## 十一、首版验收标准

首版切换完成后，至少满足以下条件：

1. 开发者可按文档启动受控浏览器并保持登录态。
2. 开发者可在不关闭浏览器的情况下完成 build + reload + 真实页验证。
3. `pnpm test:e2e` 可以作为主线自动测试入口使用。
4. 新增真实页 E2E 不再各自复制 CDP 附着逻辑。
5. 团队不再把离线路径视为主开发方案。
6. 新需求评审时，真实页验证结果成为第一证据。

## 十二、最后建议

这次路线切换最重要的原则不是“彻底删除离线能力”，而是：

**停止把离线能力当成主线真相源。**

对 TurboRender 这种强依赖宿主页面语义的插件来说，最有价值的不是构造一个看起来很像 ChatGPT 的环境，而是让开发、调试、自动测试尽可能早地附着到真实 ChatGPT 上。

因此，后续的优先级应始终保持为：

* 先稳住受控浏览器开发链路
* 再扩充 live 自动测试覆盖
* 最后处理离线路径的保留、收缩或归档
