# 本地离线 ChatGPT 开发环境需求文档（历史方案）

> 历史需求说明：这份文档记录的是旧的离线环境目标，不再代表当前主线需求。
> 现行开发主线已经切换为 `CDP + 已登录受控 Chrome + 真实 chatgpt.com`，见 [docs/plan/cdp-connected-development.md](../plan/cdp-connected-development.md)。
> 本文保留用于回顾旧设计、排查遗留脚本和理解历史权衡，不再作为新的实施基线。
> 如果本文内部的旧计划与当前 live runner 冲突，以 `pnpm test:e2e` 真实宿主路径为准。

## 1. 项目目标

为 ChatGPT TurboRender Chrome 插件开发提供**完全离线、无需网络连接**的本地开发环境，确保插件功能在可控、可重复的条件下进行开发与测试。

## 3. UI/交互需求

### 3.1 消息菜单按钮功能（复用官方实现）

**核心原则：不劫持、不复刻，让官方按钮自己工作**

插件不拦截消息菜单按钮的点击事件，让原生事件传播到 ChatGPT 自己的处理逻辑。离线模式下仅对失败的API请求返回假成功，保持UI状态同步。

```
┌─────────────────────────────────────┐
│  [复制] [喜欢] [不喜欢] [分享] [▼]  │  ← 官方按钮，自然点击
│                           更多操作   │
└─────────────────────────────────────┘
                              └─ 朗读
```

| 功能 | 策略 | 在线模式 | 离线模式 |
|------|------|----------|----------|
| **复制** | 不拦截，复用原生 | 浏览器 `clipboard` API 自然工作 | 同上，无需干预 |
| **喜欢** | 不拦截，API兜底 | 官方请求正常发送 | 拦截返回 `{"success":true}`，按钮变蓝 |
| **不喜欢** | 不拦截，API兜底 | 官方请求正常发送 | 同上 |
| **分享** | 不拦截 | 官方弹窗 | 静默忽略或弹窗提示（保持UI可用） |
| **更多操作** | 不拦截 | 官方下拉菜单展开 | 同在线 |
| **朗读** | 不拦截，可选增强 | 官方TTS/语音合成 | 浏览器 `speechSynthesis` API |

**关键约束：**
- 插件 **不绑定** 菜单按钮点击事件
- 插件 **不替换** 官方按钮的 `onclick` 处理器
- 插件仅通过 **Service Worker 拦截 API 响应** 实现离线兼容

## 4. 环境保真度与风险防控

### 4.1 核心风险：离线 vs 在线差异

| 风险点 | 影响 | 检测方案 | 缓解措施 |
|--------|------|----------|----------|
| **静态资源版本差异** | CSS/JS 缓存导致样式/行为不同 | 定期对比 `assets/` 与线上版本 hash | 自动化资源更新检测 |
| **API 响应结构差异** | 后端字段变更导致 mock 失效 | Schema 校验测试 | 从真实响应录制 fixture |
| **ChatGPT 前端更新** | DOM 结构变化导致插件选择器失效 | 真实宿主 E2E 告警 | 当前主线只接受真实 `chatgpt.com` 回归作为宿主证据 |
| **边缘情况缺失** | 仅测试"正常路径"，遗漏异常处理 | 混沌测试（注入错误） | 定期在线回归测试 |
| **性能特征差异** | 本地服务器响应过快，掩盖性能问题 | 引入人工延迟 | 性能预算在线验证 |

### 4.2 历史双轨开发策略（已不再推荐）

> 下面的双轨策略是旧方案记录。当前不再建议把离线环境作为日常主线，也不再把离线结果视为宿主兼容性证据。
> 当前主线是：`pnpm debug:mcp-chrome` 保持登录态浏览器，`pnpm reload:mcp-chrome` 重载扩展，`pnpm test:e2e -- --chat-url=...` 在真实 ChatGPT 会话上验收。

```
┌─────────────────────────────────────────────────────────────┐
│  开发阶段                                                   │
│  ├── 80% 时间：离线环境（快速迭代）                          │
│  └── 20% 时间：在线环境（验证保真度）                        │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  发布前必做                                                  │
│  └── 100% 功能在在线环境验证通过                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.3 自动化保真度检测（历史设想）

**每日同步检测**（CI 运行）：
- 对比 `shell.html` 中的关键 DOM 结构与线上版本
- 检测静态资源 hash 变化
- 验证 API 响应字段完整性

**每周在线回归**：
- 在真实 ChatGPT 运行完整 E2E 套件
- 对比离线/在线测试截图差异

> 当前仓库没有主线离线浏览器回放 spec，也没有把截图差异作为发布门禁。真实宿主 E2E 是当前唯一宿主真相源。

## 5. 技术实现需求

### 5.1 Fixture 结构

```
tests/fixtures-local/chatgpt/{fixture-id}/
├── fixture.json           # 会话元数据（ID、URL、最小轮数）
├── shell.html            # 页面 DOM 快照
├── conversation.json     # /backend-api/conversation 响应
├── synthesize.json       # /synthesize 响应
├── storage-state.json    # 登录态 Cookies
├── assets.json           # 静态资源映射清单
├── fallback.html         # 应急离线页面
└── assets/               # 本地缓存的静态资源
    ├── css/
    ├── js/
    ├── fonts/
    └── images/
```

### 5.2 网络拦截策略

| 请求类型 | 拦截策略 | 响应来源 |
|----------|----------|----------|
| `chatgpt.com/c/*` 主文档 | 重写到本地 shell.html | `shell.html` 内容 |
| `/backend-api/conversation/*` | 返回预录制 JSON | `conversation.json` |
| `/synthesize` | 返回预录制响应 | `synthesize.json` |
| 静态资源（CSS/JS/字体/图片） | 从本地 assets 目录 served | `assets/` 目录 |
| WebSocket | 模拟连接，无实际操作 | 空响应 |
| 遥测/分析请求 | 直接阻止 | 空响应 204 |

### 5.3 回放模式

- **HAR Replay**（Phase 1-2）：网络级别回放，部分资源可能 404
- **Origin Replay**（Phase 4）：完全离线，所有资源本地 served，推荐用于开发

## 8. 使用场景

### 场景 1：日常插件开发
```bash
# 启动受控浏览器
pnpm debug:mcp-chrome    # 保持控制浏览器运行
pnpm legacy:fixtures:check  # 仅检查历史 fixture bundle
```

### 场景 2：捕获新会话
```bash
# 1. 在控制浏览器中访问目标对话
# 2. 确认可访问后执行
pnpm legacy:fixtures:capture <fixture-id>

# 3. 验证捕获结果
pnpm legacy:fixtures:diagnose <fixture-id>
```

### 场景 3：离线调试
```bash
# 完全断开外网后
pnpm legacy:fixtures:check
```

### 场景 4：历史双轨开发（不再推荐）
```bash
# 旧日常开发循环（历史记录，不作为当前建议）
1. pnpm debug:mcp-chrome              # 启动控制浏览器
2. 仅把离线 bundle 当作历史资产或对照参考
3. pnpm legacy:fixtures:check         # 检查 bundle 完整性

# 当前真实宿主验证
pnpm test:e2e                         # 在真实 ChatGPT 验证
```

### 场景 5：历史发布前保真度验证（已废弃）
```bash
# 当前发布前宿主检查应直接使用真实 ChatGPT
pnpm test:e2e

# 如需维护历史 fixture，仅作为补充资产检查
pnpm legacy:fixtures:check
```
